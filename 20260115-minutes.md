## Minutes

 - Status
   - OMERO would be agnostic on the archiving side
   - OMERO could export to any storage medium
     - S3 looking interesting because of Zarr efforts
   - iRODS would store serialized 'package/file' exported from OMERO
   - OMERO could import files from that same format from anywhere, including iRODS
   - Keeping concerns and codebases and databases and status separated
   - The Zarr/S3 work is targeted at the sharing use cases
   - Decided NOT to attempt keeping OMERO storage in iRODS directly
     - Two sources of truth
     - No obvious algorithm for keeping both updated/clean/true
     - OMG my box is filling up… best practice to point at local netapp, etc.
   - 'Grown Up' answer?
     - Don't depend on a 'local storage' model
     - Rather a provided managed namespace elsewhere
     - And use import/export
     - Use OMERO as a viewer/editor rather than a storage system
   - Two use cases
     - 1) Zarr pixel buffer - for reading 'quickly' from S3
       - Pulling existing data into working memory
     - 2) Archiving to disk - aka elsewhere, not in OMERO
       - 'Save as'?
       - A) Initial Button in OMERO, serialization of OMERO internal
         - Not useful for any other software (or version)
         - But useful to 'save space' in OMERO itself
       - B) Future button is a 'better format'
         - Useful for newer versions of OMERO
         - Useful for other tools
         - But what is this format?
           - Not necessarily zarr... yet
             - It would need to be able to represent all vendor information without any loss
           - But no other real candidates either
     - OMERO has always been write once, or write none
     - Being able to wire cases 1 and 2B for a virtuous cycle
       - Save space on the OMERO box
       - Can faithfully get your data from the 'other place' / S3
     - Once you 'migrate' or 'sweep' your local storage into external storage
       - Can 'reclaim' the local storage, reducing costs/stress/maintenance
   - Phases
     - 0: Local only OMERO storage, copies of data=1	
     - 1: Local + S3 storage, copies of data=2
     - 2: Local and a script that can export elsewhere, invisible to OMERO, copies of data=2
     - 1+2: S3 storage, elsewhere (delete local), copies of data=2
     - 3 (TBD): S3 storage (references only, delete elsewhere), copies of data=1
     - Terrell words... OMERO managed (local), script (exported, not managed, invisible), S3 (referenced, visible, read-only)
